03 — Foundations

What is correctness?

Two obligations every serious algorithm must meet:

Termination

Execution finishes — no infinite loops on valid inputs.

Valid output

The result satisfies the problem specification.

Partial vs total

Partial

If it stops, it’s right

Halting implies a correct answer — but halting itself is not guaranteed.

Total

Always stops · always right

The standard we aim for in critical systems and theory.

Total correctness = termination + partial correctness

02 — Core concept

What is an Invariant?

A condition that stays true throughout execution — your anchor for correctness.

Never drifts

Each transition preserves the property — nothing “accidentally” invalidates it.

Tracks truth

You always know what remains true as the algorithm runs.

Proves goals

Combined with termination, it bridges “each step” to the final answer.

“Invariant = something that never breaks along the real execution path.”

Core definition

Formally: true before each step and after each step (when that step is taken).

05 — Loops

Loop invariant

One predicate — true at three critical moments in the loop’s life.

Before

True before the first iteration — your induction base.

→

Each step

True after every iteration — the inductive step.

→

On exit

Still true when the loop stops — tie to the postcondition.

The standard tool for proving while and for loops correct.

Think of it as a rule the loop never violates — at any iteration.

Mental model

06 — Proof scaffold

Three obligations

The standard loop-invariant proof template.

Initialization

True before the first iteration.

→

Maintenance

Preserved by each iteration.

→

Termination

Yields the final result.

1

Initialization

Show the invariant holds before the loop body runs the first time — the base of induction.

2

Maintenance

Assume it holds before an iteration; prove it still holds after — the inductive step.

3

Termination

When the loop exits, combine the invariant with the guard to prove the postcondition.

All three → the loop (and often the whole algorithm) is correct.

07 — Example

Sum of 1 to N

A classic loop — invariant stated right in the code.

sum = 0
for (i = 1 to n)
  sum = sum + i // Invariant at end of iteration i: // sum = 1 + 2 + … + i
“sum = 1 + 2 + … + i” — holds after every iteration ✓

At each step, sum is exactly the partial total — nothing missing, nothing extra.

08 — Verification

Proof of the sum loop

The three proof obligations, checked for this example.

Initialization
Before the loop, sum = 0 — the empty sum; the invariant holds trivially.
Maintenance
Each iteration adds i → sum still equals 1 + … + i; the predicate is preserved.
Termination
When i = n, we have sum = 1 + … + n — exactly the desired total.

Therefore the loop computes the correct total. ∎

Q.E.D.

10 — Perspective

Testing vs Proving

Testing checks some inputs. Proofs work for all inputs.

Testing

Limited coverage

You can run thousands of cases and still miss the one that breaks. Testing finds bugs; it doesn’t prove absence.

Invariant proof

Universal guarantee

A valid proof covers every valid input — size, edge cases, and corner cases together.

Testing shows it works sometimes. Invariants prove it always works.

Key insight

11 — Workflow

Proof process

A repeatable recipe — not guesswork.

1

Specify

State inputs, outputs, and pre/postconditions precisely.

2

Choose invariant

Pick a predicate that implies the goal when the loop stops.

3

Prove I–M–T

Establish initialization, maintenance, and termination.

4

Conclude

The algorithm satisfies its specification — systematically.

Repeatable structure — engineering, not inspiration.

12 — Pitfalls

Common mistakes

What breaks proofs — and how to avoid it.

Weak invariant
Predicate too weak to force the final result.
Incomplete
Skipping any of I, M, or T — all three matter.
Guard ≠ invariant
The guard stops the loop; the invariant explains correctness.
Tests only
“Passed tests” is not a proof.

Remember: the invariant must encode the algorithm’s story, not just restate syntax.
